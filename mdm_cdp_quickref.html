Master Data Management (MDM)
⏱ ~1 hour

What problem does MDM actually solve?

The same real-world entity (a customer, a product, a supplier) ends up with DIFFERENT, slightly inconsistent records across different systems — CRM has one spelling of a customer's name, the billing system has another, the support system has a third. MDM is the discipline of consolidating these into ONE trusted, authoritative version.

中文：同一个真实世界的实体(客户/商品/供应商)，在不同系统里往往有不一致的记录——CRM里客户名字一个写法，账单系统另一个写法，客服系统又一个写法。MDM就是把这些整合成一份可信的、权威的版本。

💡 One-liner: MDM answers the question "which record do we actually trust?" when multiple systems disagree about the same real-world thing.

The core concept: Golden Record

The "Golden Record" is the single, merged, de-duplicated, most-trustworthy version of an entity's data — built by matching and consolidating records from every source system. Example below uses a utility/energy company context (matching an SP Group-style JD).

中文："黄金记录"是把各个来源系统的记录匹配、合并、去重之后，得到的那份最可信的唯一版本。下面例子用的是能源/公用事业公司场景(对应SP Group这类JD)。

中文：图里展示3个系统(电表系统/账单系统/客服CRM)对同一个客户有不同记录——名字拼写不同、账号和电表号分开存——经过匹配+合并后得到唯一的黄金记录，把meter #和account #关联到同一个客户身上。"幸存者规则"(Survivorship rules)决定字段冲突时用哪个来源的数据。

💡 If asked "how would you build a golden record", the answer is: match records across sources (by fuzzy name/address/meter ID matching), then apply survivorship rules to decide which field wins per attribute. In a utility context, correctly linking a customer's account ID to their meter ID across systems is exactly this kind of matching problem.

Data Matching / Deduplication — the mechanics

How does a system know "John A. Smith" and "J. Smith" are the same person? Two broad approaches:

ApproachHow it worksTradeoff
Exact/Deterministic matching
精确匹配		Match on an exact key (email, national ID, phone number)
按精确字段匹配(邮箱/身份证号/电话)		Simple & fast, but misses variations (typos, format differences)
简单快速，但漏掉拼写变体/格式差异
Fuzzy/Probabilistic matching
模糊匹配		Similarity scoring algorithms (e.g. Levenshtein distance) across multiple fields, matched above a confidence threshold
用相似度算法(如编辑距离)对多字段打分，超过置信度阈值即判定匹配		Catches more true duplicates, but risks false-positive merges if the threshold is too loose
能抓到更多真实重复，但阈值太松会误判合并

中文：系统怎么知道"John A. Smith"和"J. Smith"是同一个人？两种主流方法：精确匹配(简单快但漏检多)和模糊匹配(用相似度算法，能抓到拼写变体，但需要调阈值防止误判)。

⚠️ Common interview trap: fuzzy matching threshold is a tradeoff — too loose and you merge different people into one record (false positive); too strict and you miss real duplicates (false negative). Mention this tradeoff explicitly if asked.

Cloudera Data Platform (CDP)
⏱ ~30 min

The one-sentence version

CDP is Cloudera's commercial, packaged, enterprise-supported distribution of the Hadoop ecosystem — the exact same open-source components you already know (HDFS, YARN, Hive, Spark, HBase), bundled together with management tooling, security, and vendor support.

中文：CDP就是Cloudera把你已经熟悉的开源Hadoop生态(HDFS/YARN/Hive/Spark/HBase)打包成企业级发行版，加上管理工具、安全组件和厂商支持服务。

💡 If asked "have you used CDP", the honest and strong answer: "I haven't used Cloudera's specific packaging, but I have deep hands-on experience with the exact components inside it — HDFS, YARN, Hive, Spark, HBase — so the ramp-up would mainly be learning Cloudera's management layer, not the underlying technology."

What CDP adds on top of the open-source stack you know

中文：CDP在你已经熟悉的开源组件(HDFS/YARN/Hive/Spark/HBase，绿色标注"你已经会了")之上，额外加了Cloudera Manager(集群管理GUI)和SDX(统一的安全+治理层)——这两个是CDP独有的，其余的底座都是你的强项。

💡 Only 2 things are genuinely CDP-specific: Cloudera Manager (the ops GUI) and SDX (the shared governance layer). Everything below that is exactly what you already work with daily.

Why this connects directly to your governance experience

SDX's core job — unified metadata, lineage, and access control across every engine — is conceptually the SAME problem a Data Asset Scoring Framework you built at a previous company solved (Standards/DQC/Security/Cost), just implemented as Cloudera's own product instead of something built in-house.

中文：SDX的核心工作——跨所有引擎的统一元数据/血缘/权限管理——本质上跟你之前公司自建的Data Asset Scoring Framework是同一类问题，只是CDP把它做成了产品化的功能。

💡 This is a strong thing to say in an interview: "SDX's governance goals are exactly what I already built manually in a previous role — I'd expect to pick up the CDP-specific tooling quickly since I already understand the underlying problem it solves."

CDP vs. CDH — what's the actual difference

CDH (Cloudera Distribution for Hadoop) was Cloudera's original, on-premises-only Hadoop distribution — it reached end-of-life around 2022. CDP is the successor: it merged CDH and Hortonworks' HDP into one platform that runs consistently across on-premises, hybrid, and public cloud (AWS/Azure/GCP), with a unified governance layer called SDX.

中文：CDH是Cloudera较旧的、主要只支持本地部署的Hadoop发行版，大约在2022年停止支持。CDP是取代它的新一代产品，把CDH和Hortonworks的HDP合并成一个平台，能在本地/混合云/公有云(AWS/Azure/GCP)之间保持一致运行，并新增了统一治理层SDX。
