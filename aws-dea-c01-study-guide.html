Task 1.1 — Perform Data Ingestion
Streaming vs Batch — when to use what

Streaming sources deliver data continuously in real time: Kinesis Data Streams for custom real-time applications, MSK (Managed Streaming for Apache Kafka) for Kafka-native workloads, DynamoDB Streams for capturing table changes, and AWS DMS for continuous database replication. Batch sources process data in scheduled intervals: S3 for file-based loads, Glue for large ETL, EMR for distributed processing, AppFlow for SaaS connectors, and Lambda for lightweight event-driven triggers.

Key concepts to know for exam
  • Fan-in / fan-out: fan-in aggregates multiple streams into one consumer; fan-out distributes one stream to multiple consumers (Kinesis Enhanced Fan-Out)
  • Stateful vs stateless: stateful transactions maintain context across events (session data); stateless treat each event independently
  • Replayability: Kinesis retains records for 24h–365 days, allowing replay; SQS does not natively support ordered replay
  • Throttling: when hitting DynamoDB or RDS rate limits, use exponential backoff, SQS as buffer, or Kinesis Firehose to absorb spikes
Task 1.2 — Transform and Process Data
Choosing the right transformation service

AWS Glue is the primary ETL service — serverless, scales automatically, supports PySpark, and integrates natively with S3, Redshift, and the Glue Data Catalog. Amazon EMR is for large-scale distributed processing where you need full Spark/Hadoop control or cost optimisation at very high scale. Lambda handles lightweight, event-driven transformations under 15 minutes. Redshift does in-warehouse transformations using SQL when data is already loaded.

Key patterns: using Amazon Bedrock to classify or enrich records as part of a Glue ETL job, calling Bedrock inference APIs from Lambda for per-record enrichment, and using Bedrock knowledge bases for semantic search over processed documents. Task 1.3 — Orchestrate Data Pipelines
Orchestration tools comparison

AWS Step Functions is a visual state machine — use it when you need complex branching, error handling, and retry logic across multiple AWS services with full execution history. Task 1.4 — Apply Programming Concepts
What the exam actually tests here

The exam doesn't ask you to write code — it tests your understanding of concepts. Know that Lambda concurrency limits need to be configured to prevent throttling under burst load, and that reserved concurrency sets a hard cap per function. Understand IaC: CloudFormation deploys templates declaratively, CDK generates CloudFormation using Python or TypeScript, and SAM is CloudFormation simplified for serverless functions. CI/CD for data pipelines typically means CodePipeline triggering Glue job deployment or Lambda updates. Distributed computing concepts: understand what a DAG is (Directed Acyclic Graph — Airflow, Step Functions) and why it prevents circular dependencies.

Task 2.1 — Choose the Right Data Store
Service selection — the most-tested skill in this domain

The exam will give you a scenario and ask you to pick the correct AWS data service. The core decision tree: if you need a data warehouse with SQL analytics, use Redshift. If you need NoSQL with single-digit millisecond latency at any scale, use DynamoDB. If you're building a data lake, use Lake Formation on top of S3. If you need relational OLTP, use RDS or Aurora. If you need real-time stream storage, use Kinesis Data Streams or MSK. For caching, use MemoryDB for Redis.

⭐ NEW v1.1 — Apache Iceberg and Vector Indexes

Apache Iceberg is an open table format for data lakes — it provides ACID transactions, schema evolution, and time travel on top of S3 or any object store. The exam may ask why you'd use Iceberg over Hive tables: Iceberg offers row-level updates and deletes (Hive does not), full schema evolution without table rewrites, and partition evolution without data migration. For vector indexes, HNSW (Hierarchical Navigable Small World) is approximate nearest-neighbour search optimised for speed, while IVF (Inverted File Index) clusters vectors and searches clusters first, good for very large datasets.

Task 2.2 — Data Cataloging Systems
Glue Data Catalog vs Hive Metastore vs SageMaker Catalog

AWS Glue Data Catalog is the primary technical metadata catalog for AWS data services — it stores table schemas, partition information, and connection definitions. It's the backing store for Athena, EMR, and Redshift Spectrum queries. Glue Crawlers automatically discover schemas by sampling data in S3, Redshift, or other sources. Apache Hive Metastore is the open-source equivalent, used when EMR clusters need to be compatible with on-premises Hadoop environments. Amazon SageMaker Catalog is new in v1.1 — it's a business-facing catalog for data discovery, not just technical metadata.

Task 2.3 — Manage Data Lifecycle
S3 storage tiers — know the cost vs access tradeoff

S3 Standard is for frequently accessed data — highest cost, millisecond retrieval. S3 Standard-IA is for data accessed less than once a month — lower storage cost but a retrieval fee applies. S3 Glacier Instant Retrieval matches Standard-IA pricing but is for quarterly access. S3 Glacier Flexible Retrieval takes minutes to hours — for backups. S3 Glacier Deep Archive is the cheapest at under a dollar per terabyte per month — for data you might never access again, with 12-hour retrieval. Use S3 Lifecycle Policies to automate transitions between tiers based on object age.

Task 2.4 — Data Models and Schema Evolution
Schema migration tools and design patterns

For migrating schemas between database engines, AWS SCT (Schema Conversion Tool) converts Oracle/SQL Server schemas to Aurora PostgreSQL or MySQL. AWS DMS Schema Conversion is the newer managed version integrated into DMS. For Redshift design, always consider distribution keys (determine which node owns which data) and sort keys (control query performance through data ordering). For DynamoDB, partition key design is critical — avoid hot partitions by choosing high-cardinality keys and using write sharding when needed.

Task 3.1 — Automate Data Processing
Automation services comparison

AWS Glue DataBrew is a visual data preparation tool — 250+ built-in transformations, no code required, best for data analysts preparing data for ML or reporting. Amazon Athena is serverless SQL against S3 — you pay per query by data scanned, so Parquet columnar format and partitioning dramatically cut costs. Amazon EventBridge is the event bus — it routes events between AWS services and third-party SaaS apps, and supports scheduled rules (cron or rate expressions) as a cron replacement for triggering pipelines.

Redshift (provisioned) vs Athena (serverless) — classic exam tradeoff

Choose Redshift when you have a consistent, predictable query load on structured data and need sub-second query performance — you pay for capacity whether or not you use it. Choose Athena when query patterns are ad-hoc or unpredictable — you pay only for the data scanned. The key exam scenario: if a company has a small BI team running occasional reports on S3 data, Athena is cheaper. If they run hundreds of queries per day on a stable data warehouse, Redshift gives better performance per dollar.

Task 3.3 — Maintain and Monitor Pipelines
Monitoring stack — three layers

CloudTrail records every AWS API call — who did what, when, from which IP. Use it for auditing and security forensics, not for application performance. CloudWatch Logs stores application log output — Lambda logs, Glue job logs, EMR logs all flow here. CloudWatch Logs Insights runs queries against those logs using a purpose-built query language. For log analytics at scale across many services and accounts, Amazon OpenSearch Service is the solution — use it when you need full-text search, complex aggregations, or sub-second log query response at massive log volume. Amazon Managed Grafana visualises metrics and logs from CloudWatch, Prometheus, and other sources on dashboards.

Task 3.4 — Ensure Data Quality
Data quality checks the exam expects you to know

Common data quality checks include: checking for empty fields and null values in mandatory columns, detecting type mismatches (a string where a number is expected), identifying duplicates using primary key violations, and measuring completeness (what percentage of expected records arrived). In AWS, Glue DataBrew lets you define quality rules visually and run them as part of a recipe job. Glue Data Quality (DQDL — Data Quality Definition Language) lets you write rules in code that run during Glue ETL jobs. For sampling techniques: simple random sampling is unbiased; stratified sampling preserves proportions across subgroups, important when checking quality in skewed datasets.

Tasks 4.1 & 4.2 — Authentication and Authorisation
IAM roles, policies, and Lake Formation — what to know

For the exam, understand the difference between IAM-managed policies (reusable, attached to multiple principals) and inline policies (embedded in a single principal, for permissions that shouldn't be reused). The principle of least privilege means granting only the permissions required for a specific task. AWS Lake Formation adds a permissions layer on top of S3 and the Glue Data Catalog — instead of complex S3 bucket policies and IAM, you grant column-level, row-level, and table-level permissions through Lake Formation, and it automatically generates the underlying policies. Role-based access control (RBAC) assigns permissions to roles by job function; attribute-based (ABAC) uses tags on both the resource and the principal to determine access dynamically.

Task 4.3 — Encryption and Masking
KMS, TLS, and data masking

AWS KMS manages encryption keys — CMKs (Customer Managed Keys) give you full key rotation control and cross-account encryption capability. S3 encryption options: SSE-S3 (AWS-managed keys, simplest), SSE-KMS (customer control via KMS, enables audit trail and cross-account), SSE-C (customer-provided keys, you manage key material). Encryption in transit means TLS — enforced via S3 bucket policies with a condition on aws:SecureTransport. Data masking replaces sensitive values with tokens or hashed values — use it for PII fields (names, emails, credit cards) in non-production environments. Data anonymisation is irreversible masking — removes the ability to trace back to the individual.

Tasks 4.4 & 4.5 — Audit and Privacy
CloudTrail, Macie, and data governance

CloudTrail records every API call to AWS — who called what API, from which IP, at what time, with what parameters. CloudTrail Lake is a newer managed event data store that lets you run SQL queries against CloudTrail logs without setting up Athena or S3 manually. Amazon Macie uses machine learning to automatically discover and classify sensitive data (PII) in S3 — it identifies things like passport numbers, credit cards, and health information and raises findings. Combine Macie with Lake Formation column-level security to automatically restrict access to discovered sensitive columns.

⭐ NEW v1.1 — Governance frameworks and SageMaker Catalog

The exam now tests data governance frameworks: understanding that governance covers data quality, data lineage, access control, and compliance — not just technical cataloging. SageMaker Catalog (new in v1.1) is a business-facing metadata catalog that lets data consumers discover and understand datasets without needing technical knowledge of the Glue Data Catalog. It supports projects, domain units, and custom metadata, and integrates with SageMaker Unified Studio for end-to-end data governance in ML workflows.

Q2 of 20
A data engineer wants to convert 10TB of JSON files in S3 to Parquet format to reduce Athena query costs. What is the MOST cost-effective approach?
Domain 1 — Transform
A.Load all files into Redshift, then UNLOAD to S3 as Parquet
B.Use EC2 with a custom PySpark script
C.Use an AWS Glue ETL job to read JSON and write Parquet with Snappy compression
D.Use Lambda to convert each file individually
✅ C is correct. Glue is serverless (no cluster to provision), natively reads JSON from S3 and writes Parquet, and integrates with the Glue Data Catalog for partition management. C.DynamoDB Point-in-Time Recovery
D.DynamoDB Streams with a Lambda trigger
✅ D is correct. DynamoDB Streams captures every item-level modification in near real time as an ordered stream of change records. DAX (B) is a caching layer. PITR (C) is for backup and recovery.

Q8 of 20
A Glue ETL job is failing intermittently with "out of memory" errors when processing large partitions. What is the BEST fix?

Q13 of 20
A Glue Crawler has discovered new partitions in S3 but the partitions are not visible in Athena queries. What is the MOST likely cause?
Domain 2 — Cataloging
A.The Glue Data Catalog was not updated after the crawler ran
B.Athena does not support partitioned tables
C.The S3 bucket is in a different region than Athena
D.Parquet files require a schema registry to be queried
✅ A is correct. Athena reads metadata from the Glue Data Catalog. If the Crawler completed but the catalog table's partition metadata was not refreshed (e.g., by running MSCK REPAIR TABLE in Athena or re-running the Crawler successfully), new partitions won't appear. B is false — Athena fully supports partitioned tables. C is false if both are in the same region. D is false — Parquet is natively supported.

Redshift UDFs (D) can call external ML services but add complexity.

A database has a hot partition in DynamoDB — one partition key receives 80% of all writes, causing throttling. What is the BEST mitigation?
Domain 2 — Data Store
A.Enable DynamoDB Accelerator (DAX)
B.Switch to provisioned capacity mode with auto-scaling
C.Create a Global Secondary Index on the hot key
D.Use write sharding — append a random suffix to the partition key
✅ D is correct. Write sharding distributes writes across multiple physical partitions by appending a random number (e.g., 1-10) to the partition key. This spreads the hot partition's load across 10 partitions. Your application reads from all suffixed versions to reconstruct the full dataset. DAX (A) is a read cache — it doesn't help with write throttling. Auto-scaling (B) reacts after throttling occurs, doesn't eliminate the hot partition. GSI (C) creates a separate index with the same hotness problem.
Q17 of 20
A company's Lambda function processes Kinesis records but frequently times out during spikes. What TWO changes would MOST improve throughput? (Select TWO)
Domain 1 — Programming
A.Increase the Kinesis shard count to add more parallelism
B.Increase Lambda reserved concurrency
C.Enable Kinesis Enhanced Fan-Out
D.Switch Lambda runtime from Python to Java
✅ A and B are correct. Each Kinesis shard has one Lambda consumer by default — more shards means more parallel Lambda invocations, increasing overall throughput. Increasing reserved concurrency ensures Lambda has capacity to scale without being throttled by account-level limits. Enhanced Fan-Out (C) enables multiple consumers to read at full speed, but that's for multiple distinct consumers, not increasing throughput for a single consumer. Changing runtime (D) won't solve a throughput bottleneck.

A team needs to share a Redshift data warehouse with a partner company in a different AWS account. The partner should only see specific tables and must not be able to copy data to their own S3. Which feature enables this?
Domain 4 — Governance
A.Export tables to S3 and share the bucket with cross-account permissions
B.Create an IAM role in your account that the partner can assume
C.Use Redshift Data Sharing with cross-account access and restrict the consumer cluster's IAM permissions
D.Use AWS DMS to replicate only the specified tables to the partner's Redshift
✅ C is correct. Redshift Data Sharing lets you share live, read-only access to databases, schemas, tables, and views across AWS accounts — no data movement, no ETL, no copying. Partners query live data in your cluster. You can restrict access at the table or view level. S3 export (A) copies data and doesn't enforce a no-copy restriction. IAM role (B) would give access to the entire cluster. DMS (D) copies data, which violates the no-copy requirement.

A data quality job in Glue DataBrew finds that 15% of records in a "customer_email" column contain null values. The data engineer wants to be alerted automatically if this null rate exceeds 10% in future runs. What is the BEST approach?
Domain 3 — Data Quality
A.Set up a CloudWatch alarm on the DataBrew job duration
B.Define a DataBrew data quality rule with a threshold and connect to an SNS alert on rule failure
C.Write a Lambda function to count nulls after each job
D.Enable CloudTrail on the DataBrew job to log null statistics
✅ B is correct. Glue DataBrew supports data quality rules with configurable thresholds (e.g., null rate less than 10%). When a rule fails, the job outcome can trigger an SNS notification via EventBridge. This is a declarative, code-free quality gate. CloudWatch alarm on job duration (A) measures performance, not data quality. A custom Lambda (C) works but is unnecessary when DataBrew natively supports this. CloudTrail (D) logs API calls, not data statistics.

A company wants to store credentials used by Lambda functions to connect to RDS, with automatic rotation every 30 days and auditable access logs. Which solution BEST meets these requirements?
Domain 4 — Security
A.Store credentials as Lambda environment variables encrypted with KMS
B.Store credentials in Systems Manager Parameter Store (Standard)
C.Hard-code credentials in the Lambda deployment package
D.Store in AWS Secrets Manager with automatic rotation enabled and CloudTrail logging
✅ D is correct. AWS Secrets Manager is purpose-built for credential storage with automatic rotation (Lambda rotator function), versioning, and CloudTrail logging of every access. Lambda environment variables (A) can be encrypted but offer no automatic rotation. SSM Parameter Store (B) supports SecureString but automatic rotation requires custom Lambda setup and is less feature-rich than Secrets Manager for database credentials. Hard-coding (C) is never acceptable.
