Critical detail: the JD names LlamaIndex and ADK, not LangChain

The technology stack section explicitly lists dbt, BigQuery, LlamaIndex, ADK, and n8n. LangChain is never mentioned. Despite this, LangChain remains the right starting point because the underlying concepts — agent loops, tool calling, retrieval augmented generation — are framework-agnostic, and LangChain has the most systematic teaching material available. The strategy is to build conceptual depth with LangChain, then spend a small amount of time mapping that knowledge onto LlamaIndex's different API syntax.

Why LangChain first, even though the JD says LlamaIndex
The three frameworks compared

LangChain is chain-composition-centric — its mental model is piping together a prompt, a language model, and an output parser. LlamaIndex is index-centric — its mental model is building an index over your data and querying it intelligently, which arguably fits a JD centered on a structured semantic layer even better. ADK is Google's Agent Development Kit, native to Google's ecosystem with comparatively less learning material available, optional unless an interview specifically asks about it.

What stays the same across all three frameworks

The agent loop of thinking, acting, observing, and repeating is identical regardless of framework. Tool calling and function calling work the same way. Retrieval augmented generation always follows the same four building blocks — a document loader, a text splitter, an embedding model, and a vector store. Multi-step reasoning and state management follow the same patterns. Learning these concepts once, deeply, with the best-documented framework, then porting that understanding to a different API surface is far more efficient than learning a less-documented framework from scratch.

Concept Primer — why Section 8's pattern matters for the demo

A data QA bot answering which merchant had the highest GMV last week needs to show its work — which table, which SQL, which dbt model. The content and artifact pattern splits a tool's output into two channels: content is the answer text sent back to the language model, while artifact is the raw query results or source table metadata, shown directly to the user and bypassing the model entirely. This is the exact mechanism for building trust in an internal analytics copilot, directly mapping to the JD's language about a trusted, scalable data language.

Concept Primer — LangChain vs LlamaIndex, the one-sentence answer

LangChain is chain-composition-centric, piping together a prompt, a language model, and a parser. LlamaIndex is index-centric, building an index over data and querying it intelligently. For a JD that explicitly wants natural-language querying of dbt models and BigQuery tables, LlamaIndex's index-first mental model arguably maps more naturally to indexing the semantic layer and letting an agent query it, compared to LangChain's chain-first model. Step 3 — Observability (Langfuse / Phoenix)
Fast path — Langfuse docs only, one to two days

No deployment required — reading the code examples is sufficient for interview-level fluency.

Concept Primer — Langfuse's five-object data model

A trace represents one end-to-end request, one complete user interaction. Inside a trace, a span is a unit of work with duration, such as a retrieval step or a tool call. A generation is a specific kind of span representing one language model call, carrying the model name, prompt, completion, token counts, and cost. An event is a point-in-time marker with no duration. A session groups multiple traces belonging to one multi-turn conversation. A score is a quality signal — numeric, boolean, or categorical — attached to a trace, observation, or session. This five-object model is the entire mental model behind LLM observability, and it applies whether the tool is Langfuse, Phoenix, or anything else in this category.

from langchain.tools import tool from langchain.agents import create_agent from google.cloud import bigquery @tool def query_bigquery(sql: str) -> str: """Execute a SQL query against BigQuery and return results.""" client = bigquery.Client() return str(client.query(sql).to_dataframe()) @tool def get_dbt_model_schema(model_name: str) -> str: """Read the schema definition for a dbt model.""" # parse schema.yml or manifest.json ... agent = create_agent( model=llm, tools=[query_bigquery, get_dbt_model_schema] ) # "which merchant had highest GMV last week" # → agent checks schema to understand fields # → generates SQL # → queries BigQuery # → returns answer
