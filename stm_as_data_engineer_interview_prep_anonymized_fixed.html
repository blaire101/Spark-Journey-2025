2. ETL and Data Architecture

ST3. How would you design a pipeline from multiple source systems into a data warehouse?

BothHigh probability

Suggested answerI would define source contracts, extraction frequency, keys, expected volume, and ownership. I would load source-aligned data into a staging or raw layer, then perform schema validation, completeness checks, deduplication, and business-rule validation. Transformations would populate curated or dimensional models, and publication would occur only after quality checks pass. The process should be incremental, idempotent, observable, and restartable.

ST4. What is the purpose of a staging area?

STM

Suggested answerA staging area temporarily stores extracted source data before final transformation and loading. It isolates source systems from downstream processing, supports validation and reconciliation, enables restart and replay, and provides a stable point for handling different source formats or schedules.

ST5. How do you guarantee both quantitative and qualitative data access?

STMJD wording

Suggested answerQuantitative access means the required data is complete, available, and delivered at the expected scale and frequency. Qualitative access means the data is accurate, consistent, timely, well-defined, and suitable for analysis. I monitor row counts, freshness, completeness, null rates, duplicate rates, reconciliation totals, schema validity, and business rules.

ST6. How do you model data for analysts and data scientists?

Both

Suggested answerI begin with the business process, grain, dimensions, measures, and common access patterns. I build clear fact and dimension models or curated wide tables when appropriate, document definitions, preserve lineage, use stable keys, and optimize partitions or indexes based on query patterns.

3. Databases and SQL

ST7. What is the difference between an index and a partition?

STM

Suggested answerAn index is a data structure that speeds up row lookup without scanning the full table. Partitioning physically divides a large table into segments based on a key, allowing partition pruning and easier data management.

ST8. How would you troubleshoot a slow SQL query?

BothHigh probability

Suggested answerI inspect the execution plan, identify full scans, expensive joins, large sorts, cardinality errors, and missing indexes. I verify filter selectivity, join keys, statistics, data volume, and whether functions prevent index use. Then I reduce unnecessary rows and columns, improve indexes or partitions, rewrite joins or subqueries, and test with realistic data.

ST9. What is PL/SQL and when would you use it?

STM

Suggested answerPL/SQL is Oracle’s procedural extension to SQL. It supports variables, loops, conditions, exceptions, procedures, functions, and packages. I would use it for database-side validation, stored procedures, controlled batch logic, or transactional operations close to Oracle data. My strongest language is Python, but I understand the purpose and structure of PL/SQL.

ST10. What differences matter between MySQL, MariaDB, and Oracle?

STM

Suggested answerMySQL and MariaDB are closely related open-source relational databases, while Oracle is an enterprise database with extensive features for partitioning, PL/SQL, administration, and large transactional systems. SQL is broadly similar, but functions, optimizer behavior, data types, procedural languages, sequence handling, and operational tooling differ.

Q1. How Would You Design a Reliable ETL Pipeline?

Both
I would separate the pipeline into ingestion, raw storage, validation, transformation, curated output, and serving layers. The raw layer should be immutable so data can be replayed. Each run should have a deterministic batch identifier, and writes should be idempotent to prevent duplicates after retries.

I would validate schema, volume, null rates, uniqueness, freshness, and business rules before publishing. Failures should use controlled retries with exponential backoff, while non-recoverable records should be quarantined. I would also add metrics, logs, lineage, alerts, ownership, and a clear rerun procedure.
必须说：分层、原始数据可重放、幂等、重试、数据质量、告警、可恢复。

Q2. What Is Idempotency?

Both
Idempotency means rerunning the same pipeline with the same input produces the same final result without duplicates or inconsistency. I use deterministic keys, partition overwrite, merge or upsert logic, and checkpoint metadata.

Q3. Incremental Loads

Both
I use a reliable watermark such as update time, source sequence, or CDC offset. I persist the last successful checkpoint, read only new or changed records, deduplicate by business key and latest update time, and update the target using merge or partition overwrite.

Q4. How Do You Handle Schema Changes?

Both
I classify changes as backward compatible or breaking. New nullable columns are usually safe, while changed types, renamed columns, or removed fields require coordination. I validate schemas before processing, version the contract, quarantine incompatible records, and alert producers and consumers.

Q1. Explain Your Spark Optimization Case

More likely at AS
A large aggregation job took about 75 minutes and sometimes failed with memory pressure. I used Spark UI to inspect stage duration, task distribution, shuffle size, and memory behavior. I found data skew on hot keys and high pressure from hash aggregation.

I redesigned the aggregation into two stages, applied salting, enabled adaptive query execution, broadcast a small exchange-rate dimension table, and reviewed partition counts. Runtime dropped from 75 minutes to 13 minutes, and stability improved significantly.

Q2. Spark vs MapReduce

More likely at AS
Spark supports in-memory processing, richer execution plans, SQL, streaming, and iterative workloads. MapReduce writes intermediate results to disk between stages and is generally slower for multi-stage processing.

Q3. What Causes Shuffle?

More likely at AS
Shuffle occurs when data must be redistributed across partitions, such as during groupBy, join, distinct, repartition, and many window operations. It is expensive because of network transfer, serialization, disk spill, and coordination.

Q4. Handling Data Skew

More likely at AS
I confirm skew in Spark UI by comparing task duration and partition size. Typical solutions include salting, two-stage aggregation, adaptive skew join, broadcasting small tables, filtering early, and repartitioning with a better key.

Q5. Presto vs Spark SQL

More likely at AS
Presto is mainly for interactive, low-latency SQL over distributed data sources. Spark SQL is better for heavy ETL, complex transformations, iterative processing, and broader data engineering workflows.

Q6. Hive Partitioning and Bucketing

More likely at AS
Partitioning organizes data by columns such as date or region and enables partition pruning. Bucketing hashes rows into a fixed number of files. I use partitioning for common filters and avoid over-partitioning because it creates too many small files.

Q1. Keep the Latest Record per Business Key

Both
SELECT *
FROM (
  SELECT t.*,
         ROW_NUMBER() OVER (
           PARTITION BY business_key
           ORDER BY update_time DESC
         ) AS rn
  FROM source_table t
) x
WHERE rn = 1;
PARTITION BY creates one group per entity. Descending update time puts the newest row first. Add a second ordering column when timestamps can tie.

Q2. ROW_NUMBER vs RANK

Both
ROW_NUMBER always gives unique numbers. RANK gives equal rank to ties and leaves gaps. DENSE_RANK gives equal rank without gaps.

Q3. Prevent Join Duplication

Both
Validate whether the join is one-to-one, one-to-many, or many-to-many. Deduplicate dimensions first, check key uniqueness, and compare row counts before and after the join.

Q4. Slowly Changing Dimensions

Both
Type 1 overwrites old values. Type 2 creates a new version with effective dates and a current flag, preserving history. I use Type 2 when reporting must reflect the dimension value valid at the event time.

Q1. A Production Pipeline Failed. What Do You Do?

More likely at AS
I first determine the blast radius and whether the cause is data, code, infrastructure, or an external dependency. I review alerts, logs, recent deployments, input volume, schema changes, resource usage, and upstream availability.

I then apply the safest mitigation, such as retrying from a checkpoint, isolating bad data, scaling resources, or rolling back a change. After service is restored, I complete root cause analysis and add preventive controls such as validation, monitoring, retry logic, capacity alerts, or runbook updates.
顺序：影响范围 → 数据/代码/基础设施/依赖 → 先恢复 → 根因 → 预防。

Q2. Pipeline Monitoring

Q3. Retry vs Fail Fast

More likely at AS
Retry transient network or service errors with exponential backoff. Fail fast for deterministic issues such as invalid schema, corrupted files, or broken business rules.

Q1. List vs Tuple

Both
A list is mutable. A tuple is immutable and can be used as a dictionary key when all values are hashable.

Q2. Why Use a Generator?

Both
A generator produces values lazily and avoids loading everything into memory. It is useful for large files and streaming-style processing.

Q3. Explain a Decorator

Both
def verbose(func):
    def wrapper(*args, **kwargs):
        print("Starting")
        result = func(*args, **kwargs)
        print("Finished")
        return result
    return wrapper

@verbose
def process():
    print("Processing")
@verbose is equivalent to process = verbose(process). The decorator function runs when the function is defined and returns the wrapper. Every later call to process() runs the wrapper.
