❌ BEFORE — the original (broken) query

This single-stage COUNT(DISTINCT) is what caused the entire problem. This exact query was later REPLACED — see the final production query at the end of the Action section below, which is a different query, not a patch on this one.

SELECT t.partner_remittance_provider, t.sender_country, t.currency, COUNT(t.transaction_id) AS tx_count, COUNT(DISTINCT t.sender_id) AS sender_count, SUM(t.amount * fx.fx_to_cnh) AS total_cnh, SUM(t.amount * fx.fx_to_usd) AS total_usd FROM transactions t JOIN exchange_rates fx ON t.currency = fx.currency AND t.fdate = fx.fdate WHERE t.fdate BETWEEN '20240101' AND '20241231' GROUP BY t.partner_remittance_provider, t.sender_country, t.currency;

Level 1 — Config (no code change)
Cheapest fixes. Pure Spark settings — the query stays untouched.

3.Fix 1: Tune Spark configuration & enable AQE — try this FIRST, no code change

Always try config first — if this alone solves it, you never have to touch the query logic at all.

--num-executors 100 --executor-cores 4 --executor-memory 6G --driver-memory 4G SET spark.sql.shuffle.partitions = 2000; SET spark.dynamicAllocation.enabled=true; SET spark.dynamicAllocation.maxExecutors=120; SET spark.dynamicAllocation.minExecutors=5; SET spark.sql.adaptive.enabled = true; SET spark.sql.adaptive.coalescePartitions.enabled = true; SET spark.sql.adaptive.skewJoin.enabled = true; SET spark.sql.adaptive.skewedPartitionThresholdInBytes = 64MB; SET spark.sql.adaptive.skewedPartitionFactor = 5; SET spark.sql.autoBroadcastJoinThreshold = 10MB;

Coalesce partitionsMerges small shuffle partitions at runtime

💡 2000 shuffle partitions (up from default 200) gave finer-grained skew detection, and AQE cleaned up the MILD skew automatically — but the two EXTREME hot keys were still too large for AQE to split alone.

-- Stage 1: dedup rows to one row per (partner_remittance_provider, sender_country, currency, sender_id) WITH dedup AS ( SELECT t.partner_remittance_provider, t.sender_country, t.currency, t.sender_id FROM transactions t WHERE t.fdate BETWEEN '20240101' AND '20241231' GROUP BY t.partner_remittance_provider, t.sender_country, t.currency, t.sender_id ) -- Stage 2: count deduplicated sender_id rows per dimension (exact distinct count) SELECT d.partner_remittance_provider, d.sender_country, d.currency, COUNT(*) AS sender_count FROM dedup d GROUP BY d.partner_remittance_provider, d.sender_country, d.currency;
💡 This converts one unspillable giant HashSet into many small spillable groups — runtime became stable, but the 2 extreme hot keys were STILL the slowest part of every run.

6.✅ AFTER — Fix 4: Targeted salting — the FINAL production query

The last resort — reserved for the SPECIFIC extreme keys that survived every earlier, less invasive fix. This is a COMPLETELY DIFFERENT query from the original 'BEFORE' query shown in the Situation section above — not a small patch, a full rewrite of the aggregation logic.

-- A. Dedup exchange rates first (avoid duplicate join fan-out) WITH fx_dedup AS ( SELECT currency, fdate, fx_to_cnh, fx_to_usd FROM ( SELECT currency, fdate, fx_to_cnh, fx_to_usd, ROW_NUMBER() OVER (PARTITION BY currency, fdate ORDER BY fdate DESC) AS rn FROM exchange_rates WHERE fdate BETWEEN '20240101' AND '20241231' ) d WHERE rn = 1 ), -- B. Salt only the 2 known hot keys; everyone else goes to fixed bucket 8 salted_txn AS ( SELECT t.partner_remittance_provider, t.sender_country, t.currency, t.sender_id, t.transaction_id, t.amount, t.fdate, CASE WHEN (t.sender_country = 'US' AND t.partner_remittance_provider = 'Wise') OR (t.sender_country = 'KR' AND t.partner_remittance_provider = 'PandaRemit') THEN (HASH(t.sender_id) & 2147483647) % 8 -- hot keys → 8 buckets ELSE 8 -- everyone else → fixed bucket 8 END AS salt_bucket FROM transactions t WHERE t.fdate BETWEEN '20240101' AND '20241231' ), -- C. Stage 1: aggregate per dimension + salt bucket (exact dedup within bucket) agg_stage1 AS ( SELECT s.partner_remittance_provider, s.sender_country, s.currency, s.salt_bucket, COUNT(s.transaction_id) AS tx_cnt_bucket, COUNT(DISTINCT s.sender_id) AS sender_cnt_bucket, SUM(s.amount * fx.fx_to_cnh) AS total_cnh_bucket, SUM(s.amount * fx.fx_to_usd) AS total_usd_bucket FROM salted_txn s JOIN fx_dedup fx ON s.currency = fx.currency AND s.fdate = fx.fdate GROUP BY s.partner_remittance_provider, s.sender_country, s.currency, s.salt_bucket ) -- D. Stage 2: sum buckets back to original dimensions (still exact) SELECT partner_remittance_provider, sender_country, currency, SUM(tx_cnt_bucket) AS tx_count, SUM(sender_cnt_bucket) AS sender_count, SUM(total_cnh_bucket) AS total_cnh, SUM(total_usd_bucket) AS total_usd, ROUND(SUM(total_cnh_bucket) / NULLIF(SUM(tx_cnt_bucket), 0), 2) AS avg_txn_amt_cnh FROM agg_stage1 GROUP BY partner_remittance_provider, sender_country, currency;
💡 Because the salt only depends on sender_id, the same sender ALWAYS lands in the same bucket — summing bucket counts gives an EXACT distinct count, not an approximation.

7.Related technique: general JOIN salting (for future unknown hot keys)

A reusable pattern for future incidents where the hot key isn't known ahead of time — not used in this specific fix, but documented for the team.

-- Large table: deterministic salt for every row, based on currency+fdate WITH salted_txn AS ( SELECT t.*, (HASH(t.currency, t.fdate) & 2147483647) % 8 AS salt FROM transactions t WHERE t.fdate BETWEEN '20240101' AND '20241231' ), -- Small table: replicate exchange_rates 8 times, salt = 0..7 salted_fx AS ( SELECT fx.*, CAST(r.id AS INT) AS salt FROM exchange_rates fx CROSS JOIN (SELECT CAST(id AS INT) AS id FROM range(8)) r ) -- Join on (currency, fdate, salt) instead of just (currency, fdate) SELECT s.partner_remittance_provider, s.sender_country, s.currency, COUNT(s.transaction_id) AS tx_count, SUM(s.amount * f.fx_to_cnh) AS total_cnh FROM salted_txn s JOIN salted_fx f ON s.currency = f.currency AND s.fdate = f.fdate AND s.salt = f.salt GROUP BY s.partner_remittance_provider, s.sender_country, s.currency;
💡 Trade-off vs targeted salting: simpler, no need to know hot keys in advance, but replicates the WHOLE small table N times.

8.Related technique: targeted JOIN salting (when the hot key IS known)

More efficient than the general pattern above — use this variant once you've identified the SPECIFIC hot key, without replicating the ENTIRE small table.

-- Large table: only the known hot key gets a randomized salt CASE WHEN f.merchant_id = 123 THEN CAST(FLOOR(RAND() * 8) AS INT) + 1 -- 1..8 ELSE 0 END AS salt -- Small table: hot key replicated 1-8, everyone else stays at 0 SELECT d.*, 0 AS salt FROM dim_merchant d WHERE d.merchant_id <> 123 UNION ALL SELECT d.*, s.salt FROM dim_merchant d JOIN (SELECT 1 AS salt UNION ALL SELECT 2 UNION ALL SELECT 3 UNION ALL SELECT 4 UNION ALL SELECT 5 UNION ALL SELECT 6 UNION ALL SELECT 7 UNION ALL SELECT 8) s ON d.merchant_id = 123
💡 Trade-off vs the general pattern: only the ONE hot row gets replicated 8 times, not the whole table — more efficient once you know exactly which key is hot.

Data Skew — Causes & Problems
Understand WHY skew happens and WHAT it breaks, before touching any SQL.

What is data skew?

Data skew means rows are NOT evenly distributed across the GROUP BY / shuffle key — a small number of key values (like 'x' above) hold a disproportionate share of the total rows.

Common causes of data skew — 5 typical patterns

Skew almost always traces back to a real, non-random property of the business data — not bad luck.

Detail
CauseExample
Power-law user/customer behaviorA handful of VIP accounts or bot accounts generate most transactions
Default / placeholder valuesNULL, 0, or 'UNKNOWN' used as a fallback dominates the join or group key
Popular product / hot categoryOne best-selling SKU or trending category dwarfs all others
Time-based clusteringMost rows are from 'today' in an incrementally-loaded table
Upstream join fan-outA join earlier in the pipeline duplicated rows for one specific key
💡 Diagnosing the CAUSE matters — a NULL-dominated join key is fixed differently (filter or isolate NULLs) than a genuinely hot VIP customer (salting).

What problems does skew actually cause?

Four concrete failure modes, from mild to catastrophic.

💡 The business impact is usually #1 and #3 combined: the job takes far longer than the DATA VOLUME alone would suggest, because parallelism collapses down to whatever the hot key's single task is doing.

Why multi-COUNT(DISTINCT) makes it worse specifically

With two distinct dimensions (c and d) in the same GROUP BY, the hot key's single task must hold TWO separate deduplication structures in memory at once — for the SAME oversized partition.

💡 This is the same underlying failure as the HashAggregate OOM in the SLA case study — just doubled, because c and d each need their own HashSet on the same hot task.

What does a task's memory actually look like, precisely?

A task doesn't process just ONE key — it processes a whole batch of keys. Internally it holds one HashMap, where EACH key gets its own aggregation buffer, and each COUNT(DISTINCT) column gets its own HashSet inside that buffer.

💡 The formula is: total HashSets per task = (distinct GROUP BY keys landing on that task) × (number of COUNT(DISTINCT) expressions). With 3 keys and 2 distinct columns, that's 6 — but memory pressure comes almost entirely from the hot key's 2 oversized ones, not the other small ones.

Level 2 — Split into single-dimension subqueries
Only reached because Level 1 showed real slowness or OOM on large data.

Split into single-dimension subqueries, then JOIN

Each subquery handles exactly ONE aggregation dimension, so no single task ever needs to hold two HashSets at once.

SELECT t1.a, t1.sum_b, t2.cnt_c, t3.cnt_d FROM (SELECT a, SUM(b) AS sum_b FROM test GROUP BY a) t1 JOIN (SELECT a, COUNT(DISTINCT c) AS cnt_c FROM test GROUP BY a) t2 ON t1.a = t2.a JOIN (SELECT a, COUNT(DISTINCT d) AS cnt_d FROM test GROUP BY a) t3 ON t1.a = t3.a;

Alternative: UNION ALL with a tag column (single scan)

Tags each row with which dimension it belongs to, scans the table only once, then filters per-dimension inside the aggregate functions.

SELECT a, SUM(CASE WHEN tag=0 THEN b END) AS sum_b, COUNT(DISTINCT CASE WHEN tag=1 THEN c END) AS cnt_c, COUNT(DISTINCT CASE WHEN tag=2 THEN d END) AS cnt_d FROM ( SELECT a, b, NULL AS c, NULL AS d, 0 AS tag FROM test UNION ALL SELECT a, NULL, c, NULL, 1 FROM test UNION ALL SELECT a, NULL, NULL, d, 2 FROM test ) unioned GROUP BY a;
💡 One scan instead of three, but the intermediate UNION ALL result is 3x the row count. Usually split+JOIN is easier for the optimizer to parallelize well.

Level 3 — Data skew: salting the hot key
Only needed when a specific key is disproportionately large.

When GROUP BY a, c alone is enough — and when it isn't

Grouping by the composite key (a, c) shuffles by hash(a, c), not just hash(a) — so it often self-distributes even when 'a' is a hot key. The deciding factor is c's cardinality WITHIN that hot key.

Detail
ScenarioDistinct c under hot key 'x'Is (a,c) shuffle skewed?
This dataset4 distinct values (c1,c2,c3,c4)Mild — spreads across 4 shuffle partitions, not 1
Worse caseOnly 1 distinct value repeated 1000s of timesYes — every row still funnels into ONE partition
💡 In THIS dataset, GROUP BY a,c alone already spreads key 'x' across 4 partitions instead of 1 — a real improvement even before adding any salt.

Salting — for when (a,c) itself is still skewed

Add a deterministic hash-based salt bucket to spread the dedup work across more partitions, then sum the per-bucket counts back together.

WITH salted AS ( SELECT a, b, c, d, (HASH(c) & 2147483647) % 8 AS salt_c, (HASH(d) & 2147483647) % 8 AS salt_d FROM test ), c_dedup AS ( SELECT DISTINCT a, salt_c, c FROM salted ), cnt_c AS ( SELECT a, SUM(cnt) AS cnt_c FROM ( SELECT a, salt_c, COUNT(*) AS cnt FROM c_dedup GROUP BY a, salt_c ) x GROUP BY a ), d_dedup AS ( SELECT DISTINCT a, salt_d, d FROM salted ), cnt_d AS ( SELECT a, SUM(cnt) AS cnt_d FROM ( SELECT a, salt_d, COUNT(*) AS cnt FROM d_dedup GROUP BY a, salt_d ) x GROUP BY a ) SELECT a, cnt_c, cnt_d FROM cnt_c JOIN cnt_d USING (a);

The leanest correct version — piggyback SUM(b) onto the dedup scan

Instead of a separate scan just for SUM(b), compute it as a byproduct of the (a,c) aggregation — saving one full table scan compared to the 3-way split.

WITH q1 AS ( SELECT a, c, SUM(b) AS sum_b_ac FROM test GROUP BY a, c ), q2 AS ( SELECT a, d FROM test GROUP BY a, d -- no need to recompute sum(b) here ), q1_agg AS ( SELECT a, SUM(sum_b_ac) AS sum_b, COUNT(*) AS cnt_c FROM q1 GROUP BY a ), q2_agg AS ( SELECT a, COUNT(*) AS cnt_d FROM q2 GROUP BY a ) SELECT q1_agg.a, q1_agg.sum_b, q1_agg.cnt_c, q2_agg.cnt_d FROM q1_agg JOIN q2_agg ON q1_agg.a = q2_agg.a;

📘 Basics

Query structure & clauses

The core clause order every query follows, plus common NULL-handling functions.

-- NULL handling SELECT IFNULL(score, 0) AS score_val FROM student; SELECT COALESCE(score1, score2, 0) AS score_val FROM student; -- COALESCE is more versatile — accepts multiple fallback values
💡 COALESCE > IFNULL when you need multiple fallback values, not just one.

Finding duplicates

COUNT(*) vs COUNT(column) — a subtle but important distinction.

SELECT name, COUNT(*), COUNT(student_id) -- COUNT(column) only counts non-NULL values FROM students GROUP BY name HAVING COUNT(*) > 1;
💡 COUNT(*) counts all rows. COUNT(column) skips NULLs in that column.

Left Join — Anti-join pattern

Find rows in A whose key does NOT exist in B.

SELECT a.* FROM a LEFT JOIN b ON a.key = b.key WHERE b.key IS NULL;
💡 This is the standard 'anti-join' pattern — LEFT JOIN + WHERE ... IS NULL.

Joins & Subqueries overview

Quick reference for the join family.

INNER JOIN -- only matching rows on both sides LEFT JOIN -- all rows from left, matched or NULL from right RIGHT JOIN -- all rows from right, matched or NULL from left SELF JOIN -- a table joined to itself CROSS JOIN -- every combination (cartesian product)

Window function basics

The building blocks: PARTITION BY, ORDER BY, and the functions that use them.

-- Ranking ROW_NUMBER() -- unique rank, no ties RANK() -- ties share rank, next rank skips DENSE_RANK() -- ties share rank, no skip -- Analytics SUM(), AVG(), MIN(), MAX() OVER (...) -- Positional LAG() -- access the PREVIOUS row LEAD() -- access the NEXT row FIRST_VALUE(), LAST_VALUE() -- Example ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date) LAG(txn_date) OVER (PARTITION BY customer_id ORDER BY txn_date)

🔄 Retention

Rolling vs Cohort retention — the concept

The key conceptual difference before writing any SQL.

💡 Rolling = 'check every day who came back.' Cohort = 'take a batch of new users and see how many return.'

A) Rolling 30-day retention

Gap-based approach — measures the gap between EACH pair of consecutive transactions.

Sample data
seller_idtransaction_datenote
2012024-01-02
2012024-01-20← return within 30 days ✅
2012024-02-02← return within 30 days ✅
2022024-01-02
2022024-01-25← return within 30 days ✅
2032024-01-02← no return ❌
-- Step 1: previous date + gap SELECT seller_id, transaction_date, LAG(transaction_date) OVER (PARTITION BY seller_id ORDER BY transaction_date) AS prev_date, DATEDIFF(transaction_date, LAG(transaction_date) OVER (PARTITION BY seller_id ORDER BY transaction_date)) AS gap_days FROM transactions; -- Step 2: daily retention rate , daily AS ( SELECT transaction_date, COUNT(*) AS total_active, COUNT(prev_date) AS eligible_repeaters, SUM(CASE WHEN gap_days <= 30 THEN 1 ELSE 0 END) AS retained FROM with_gap GROUP BY transaction_date ) SELECT transaction_date, total_active, eligible_repeaters, retained, CASE WHEN eligible_repeaters = 0 THEN NULL ELSE ROUND(retained * 1.0 / eligible_repeaters, 2) END AS rate_among_eligible FROM daily ORDER BY transaction_date;
💡 eligible_repeaters excludes first-time transactions (they have no previous date to measure a gap from).

B) Cohort 30-day retention

Four-step process: anchor date, candidate returns, per-seller flag, cohort-level rate.

Sample data

2012024-01-02
2012024-01-20← within 30 days ✅

2022024-01-02
2022024-01-25← within 30 days ✅
2032024-01-02← no return ❌
-- Step 1: first transaction per seller (cohort anchor) WITH first_txn AS ( SELECT seller_id, MIN(transaction_date) AS first_date FROM transactions GROUP BY seller_id ), -- Step 2: candidate returns within (0, 30] days after first_date cand_returns AS ( SELECT f.seller_id, f.first_date, t.transaction_date AS return_date, DATEDIFF(t.transaction_date, f.first_date) AS gap_days FROM first_txn f LEFT JOIN transactions t ON t.seller_id = f.seller_id AND DATEDIFF(t.transaction_date, f.first_date) BETWEEN 1 AND 30 ), -- Step 3: collapse to per-seller retained flag (0/1) flags AS ( SELECT seller_id, first_date, CASE WHEN COUNT(return_date) > 0 THEN 1 ELSE 0 END AS retained_30d FROM cand_returns GROUP BY seller_id, first_date ) -- Step 4: final cohort retention rate SELECT first_date AS cohort_date, COUNT(*) AS cohort_size, SUM(retained_30d) AS retained_30d, ROUND(SUM(retained_30d) * 1.0 / COUNT(*), 2) AS retention_rate_30d FROM flags GROUP BY first_date ORDER BY cohort_date;
