The core mental shift

Spark is batch-first, with streaming bolted on as micro-batch simulation. Flink is streaming-first, where batch is just a special case of a bounded stream. Once this distinction clicks, 80% of Flink's design decisions stop feeling arbitrary.

#QuestionAnswer
1What triggers Flink execution?env.execute() for DataStream, or any SQL INSERT INTO. Unlike Spark's collect(), this usually runs forever
2What is a Flink Job?The whole pipeline submitted via execute() — equivalent to a Spark Job, but typically long-running, not one-shot
3What is Operator Chaining?Flink fuses adjacent operators (e.g. map→filter) into one task to avoid serialization overhead — Spark has no equivalent within a single Stage
4What is a Slot?A TaskManager's unit of resource allocation — one slot runs one subtask or one operator chain. Roughly maps to "one core's worth of executor capacity," but slots are fixed at TaskManager startup, not dynamic
Chapter 3 · Flink SQL Core Syntax (priority)
This is what to actually memorize

Since your team develops in Flink SQL, this chapter is the highest priority. The five scenarios below cover the syntax you'll write daily.

3.1 Creating a table — Spark vs Flink
SparkFlink SQL
df = spark.read.format("kafka").option(...).load()
runs once, returns a static snapshot		 CREATE TABLE ... WITH ('connector'='kafka', ...)
declares a continuous source, never "loads" once
CREATE TABLE orders ( order_id STRING, user_id STRING, amount DECIMAL(10,2), order_time TIMESTAMP(3), WATERMARK FOR order_time AS order_time - INTERVAL '5' SECOND ) WITH ( 'connector' = 'kafka', 'topic' = 'orders', 'properties.bootstrap.servers' = 'localhost:9092', 'format' = 'json', 'scan.startup.mode' = 'latest-offset' );

The WATERMARK FOR line has no Spark SQL equivalent. It's mandatory thinking for any streaming aggregation — it tells Flink when it's safe to close a window.

3.2 Streaming aggregation — Spark vs Flink
-- Spark equivalent runs once on a static dataset: -- df.groupBy("user_id").agg(sum("amount")) -- This Flink query runs continuously, re-emitting as new orders arrive SELECT user_id, TUMBLE_START(order_time, INTERVAL '5' MINUTE) AS window_start, SUM(amount) AS total_amount, COUNT(*) AS order_count FROM orders GROUP BY user_id, TUMBLE(order_time, INTERVAL '5' MINUTE);
This is the single biggest behavioral shift: one snapshot vs an endless stream of results
Why you can't GROUP BY user_id without a window

Spark's GROUP BY runs once on a static dataset. Flink's stream is unbounded — without a window, GROUP BY becomes a running total from job start to now, never emitting periodic results, and state grows without bound. So Flink SQL aggregation almost always needs a window.

3.3 Temporal table join (syntax Spark has nothing like)
SELECT o.order_id, o.amount, u.user_tier FROM orders AS o JOIN user_profile FOR SYSTEM_TIME AS OF o.order_time AS u ON o.user_id = u.user_id;
Why this syntax exists

Scenario: an order stream joined to a user tier dimension table, but tiers change — regular member today, VIP tomorrow. A regular JOIN picks up the "current" tier, not the tier at order time. If the user later upgrades, historical orders' JOIN results would change too — usually wrong. FOR SYSTEM_TIME AS OF resolves this by looking up the dimension table's snapshot value as of the order's event time. This is a temporal table join — Spark has no native syntax for it and requires significant custom logic to simulate.

Chapter 4 · Watermark (the one genuinely new concept)
Spend real time here

Spark Structured Streaming has a simplified withWatermark(), but because it operates on micro-batches the semantics are coarser. A true event-at-a-time engine like Flink needs Watermark to be a first-class, per-record concept.

Spark checks lateness once per micro-batch; Flink tracks it continuously, record by record
Watermark works off Event Time, not Processing Time
#QuestionAnswer
25What is Event Time?The timestamp embedded in the data — when the event actually happened. Almost always the correct semantics
26What is Processing Time?The wall-clock time Flink processes the record. Fast and simple, but wrong for out-of-order data
27What is a Watermark?A signal stating "no event earlier than this is expected." It's how Flink decides a window is safe to close
29What happens to data arriving after the watermark passes?Dropped by default. Configure allowed lateness to keep updating the result for a grace period, or route to a side output to avoid silent data loss
Watermark syntax, decoded
WATERMARK FOR order_time AS order_time - INTERVAL '5' SECOND

order_time is the event time field, and minus 5 seconds is the tolerated out-of-order delay — meaning events up to 5 seconds late are still accepted. Past that, the window closes and computes. Don't guess this number — base it on the measured out-of-order distribution in production, covering the p99 lateness.

allowedLateness extends acceptance past the watermark; anything later goes to a side output instead of vanishing

Mapping to your Iceberg experience

Your Hive-to-Iceberg migration at Xiaomi was batch writes into Iceberg tables. Flink writing to Iceberg is the same table, but continuous — each Checkpoint commits a new snapshot. Downstream Athena or Spark reads of that table are completely unaware which engine wrote it. Supporting both streaming writes and batch queries is one of Iceberg's core selling points.

Chapter 8 · Skew & Backpressure (your Spark playbook, ported)
You already know this pattern from your Tencent 75min→13min optimization

Below is the same playbook written in Flink terms.

Spark conceptFlink equivalentNotes
Skew on groupByKey/reduceByKeySkew on keyBySame root cause: one key gets disproportionate volume
Two-stage aggregation (local + global)Local aggregation + keyBy mergeIdentical pattern: salted pre-aggregation, merged in a second stage
SaltingSame techniqueAppend random/hash suffix to hot key, split across N parallel subtasks
AQE runtime skew splitNo direct equivalentA real capability gap — Flink doesn't replan partitions at runtime the way Spark AQE does

-- Stage 1: salted pre-aggregation WITH salted AS ( SELECT CONCAT(CAST(CAST(RAND() * 10 AS INT) AS STRING), '_', merchant_id) AS salted_key, merchant_id, amount FROM transactions ), partial_agg AS ( SELECT salted_key, merchant_id, SUM(amount) AS partial_sum FROM salted GROUP BY salted_key, merchant_id ) -- Stage 2: merge after removing salt SELECT merchant_id, SUM(partial_sum) AS total_amount FROM partial_agg GROUP BY merchant_id;
Why Flink has no AQE-style runtime replanning

Spark AQE can replan at runtime because Spark is batch — all the data already exists, so mid-execution it can see actual partition size statistics and dynamically merge or split. Flink is streaming — data never "already exists," it never has a complete statistical picture to replan against. The countermeasures for hot keys are therefore: pre-emptive salting (most common), increasing the operator's parallelism, or manually redistributing load with rebalance/rescale operators.

第一部分 · 基础入门
Flink 是什么

Flink 是一个分布式流处理引擎，核心定位是"以流为本"——它把批处理看作流处理的一个特例（有限流），而不是像 Spark 那样把流处理看作批处理的微小切片（微批处理）。这个设计哲学的区别决定了 Flink 在低延迟场景下天然占优势，因为它是真正的逐条事件处理，不需要等待凑够一批数据。

整体架构：JobManager 和 TaskManager

Flink 集群分两种角色。JobManager 是大脑，负责接收作业、调度任务、协调 Checkpoint、处理故障恢复——一个集群通常只有一个活跃的 JobManager（配合高可用部署可以有备用）。TaskManager 是干活的工人，真正执行算子逻辑、管理本地状态、和上下游交换数据。每个 TaskManager 有若干个 Slot，决定了它能并行跑多少个子任务。

JobManager 负责调度，TaskManager 实际执行任务，每个 Slot 跑一个并行子任务
DataStream API vs Table API/SQL

DataStream API 是最底层、最灵活的编程接口，用 Java 或 Scala 写，可以精确控制每个算子的行为，适合复杂的自定义逻辑。Table API 和 Flink SQL 是更高层的声明式接口，写法接近标准 SQL，开发效率更高，但底层最终也会被优化成 DataStream 的执行计划。生产环境中，简单的聚合和过滤逻辑优先用 SQL，复杂的状态管理和自定义算子用 DataStream API。

部署模式：Standalone / YARN / Kubernetes

Standalone 模式是 Flink 自带的集群管理方式，适合学习和小规模测试，不依赖外部资源调度器。YARN 模式把 Flink 跑在 Hadoop 生态的 YARN 上，是很多传统大数据团队的标准选择，能和 Hive、Spark 共享同一套资源池。Kubernetes 模式是现在的主流趋势，原生支持容器化部署，弹性伸缩能力更强，也更容易和云原生的 CI/CD 流程整合。

作业提交模式：Session / Per-Job / Application

Session 模式是先启动一个常驻集群，多个作业共享这一个集群资源，适合频繁提交的小作业，但隔离性差，一个作业的问题可能影响其他作业。Per-Job 模式为每个作业单独启动一个专属集群，作业结束集群也随之销毁，隔离性好但启动开销大。Application 模式是较新的推荐方式，把用户代码打包进集群镜像直接运行，main 方法在集群里执行而不是在客户端，减少了客户端到集群之间的网络传输和单点压力。

基本编程模型：Source → Transformation → Sink

任何 Flink 作业的结构都遵循这个三段式：Source 负责接入数据，常见的有 Kafka、文件系统、自定义数据源；Transformation 是中间的处理逻辑，包括 map、filter、keyBy、window、join 等算子；Sink 负责把结果写出去，常见的有 Kafka、数据库、文件系统、打印到控制台。理解这个模型之后，看任何 Flink 代码都能快速定位数据从哪来、怎么变、到哪去。

// 一个最简单的 Flink 作业：读 Kafka，统计每个用户的点击次数 StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment(); DataStream<ClickEvent> clicks = env .fromSource(kafkaSource, watermarkStrategy, "clicks") // Source .keyBy(ClickEvent::getUserId) // 按用户分组 .window(TumblingEventTimeWindows.of(Time.minutes(5))) // 5分钟滚动窗口 .sum("count"); // Transformation clicks.sinkTo(kafkaSink); // Sink：写回 Kafka env.execute("user-click-count");

第二部分 · 进阶调优
状态后端：HashMapStateBackend vs RocksDBStateBackend

HashMapStateBackend 把状态完全存在 JVM 堆内存里，读写速度最快，但受限于内存大小，适合状态数据量不大的作业。RocksDBStateBackend 把状态存在本地磁盘的 RocksDB 数据库里，只在内存里保留一部分热数据作为缓存，能支持远超内存容量的超大状态，代价是读写比纯内存模式慢一些。生产环境处理大状态作业——比如需要维护几亿用户的历史行为——几乎都用 RocksDB。

Checkpoint 与 Savepoint 的区别

Checkpoint 是 Flink 自动周期性触发的状态快照，专门用于故障自动恢复，文件生命周期由 Flink 自己管理，作业正常运行时旧的 Checkpoint 会被自动清理。Savepoint 是用户手动触发的状态快照，格式更稳定、更适合跨版本兼容，主要用途是有计划的作业升级、调整并行度、或者迁移到新集群——本质上是给运维人员用的"存档点"，不会被自动清理，需要手动管理生命周期。

两阶段提交：端到端 Exactly-Once 的实现原理

要做到端到端的精确一次，光靠 Checkpoint 保证内部状态一致还不够，写到外部系统（比如 Kafka）的结果也必须保证不重不漏。Flink 用两阶段提交协议实现这一点：第一阶段，Sink 把数据写成一个"预提交"的事务，绑定到当前的 Checkpoint；第二阶段，只有当这个 Checkpoint 被确认成功完成后，Sink 才会真正提交这个事务，让数据对外可见。如果作业失败重启，所有未完成第二阶段的预提交事务都会被回滚，从而避免重复写入。

两阶段提交保证：要么完整提交，要么彻底回滚，不会出现中间状态
并行度调优：parallelism 怎么设置

并行度决定一个算子能同时跑多少个并行实例。设置过低，无法充分利用集群资源，吞吐量上不去；设置过高，会产生过多的小任务，增加调度和网络通信开销，也可能导致下游算子来不及处理而产生反压。常见做法是先用 Source 的天然并行度（比如 Kafka 分区数）作为基准，再结合 Flink Web UI 观察各算子的繁忙程度，针对性地给瓶颈算子单独设置更高的并行度，而不是无脑给所有算子设同一个数值。

反压（Backpressure）的产生和定位

反压本质是下游处理能力跟不上上游产生数据的速度，Flink 的网络栈会自动把这种压力沿着数据流向上游传导，最终表现为 Source 也被迫减速读取。定位反压要用 Flink Web UI 的反压监控面板，从下游往上游依次排查，第一个显示"高反压"的算子往往才是真正的瓶颈所在——它后面的算子之所以也显示反压，是被它拖累的，不要被表面现象误导而去优化错误的算子。

反压排查的关键：被拖累的算子不是问题根源，最下游那个才是
State TTL：状态过期清理

长期运行的流作业如果给每个 key 维护状态却从不清理，状态会无限增长，最终拖垮 Checkpoint 性能甚至撑爆磁盘。State TTL 让你给状态描述符配置一个存活时间，比如 7 天未更新就自动失效清除。需要注意更新策略的选择：OnCreateAndWrite 表示只在创建和写入时刷新过期时间，OnReadAndWrite 表示连读取也会刷新过期时间——选错策略可能导致状态该清的时候没清，或者清得太早影响业务正确性。

增量 Checkpoint：应对大状态作业

普通的全量 Checkpoint 每次都要把当前全部状态完整写一遍，状态越大，Checkpoint 耗时越长，甚至可能超过 Checkpoint 间隔导致排队失败。增量 Checkpoint（配合 RocksDB 状态后端使用）只持久化相对上一次 Checkpoint 的增量变化部分，大幅缩短单次 Checkpoint 的耗时，这是处理超大状态作业——比如需要维护几个月用户行为历史的场景——几乎必须开启的优化项。

Apache Flink

Core Concepts

The building blocks you need before discussing real-time architecture.

Requires internet. Open in Chrome for audio.
Stream processing fundamentals
Stream vs Batch — unified API

Flink treats batch as a special case of streaming — a bounded stream. The same DataStream API runs both unbounded real-time streams and bounded historical datasets. This is different from Spark, where batch and streaming are conceptually separate engines bolted together with micro-batching.

Event Time vs Processing Time

Event time is the timestamp embedded in the data itself — when the event actually happened. Processing time is when Flink happens to process that event. In distributed systems, events arrive out of order due to network delay, retries, or upstream batching. Using event time with watermarks lets Flink produce correct results regardless of when data physically arrives.

Watermark

A watermark is Flink's mechanism for saying "I believe no more events with a timestamp earlier than this will arrive." It is how Flink knows when a time window is safe to close and emit results, without waiting forever for late data. A common pattern is watermark equals max event time seen so far minus a fixed delay, such as 5 minutes, to tolerate some out-of-order arrival.

State and fault tolerance
State

State is data Flink remembers across events — for example, a running sum per user, or the last seen timestamp per key. Flink state is local to each parallel task and stored either in memory or on disk via RocksDB for state too large to fit in memory.

Checkpoint

A checkpoint is a consistent snapshot of the entire job's state at a point in time, written to durable storage such as S3 or HDFS. If a task fails, Flink restarts from the last successful checkpoint rather than from scratch. Checkpoints are coordinated using the Chandy-Lamport algorithm via barriers injected into the stream.

Exactly-once semantics

Flink achieves exactly-once processing by combining checkpointing with transactional sinks. On recovery, Flink replays events from the last checkpoint and the sink either commits or rolls back the transaction atomically, so duplicate writes are prevented even though the same events may be reprocessed internally.

Windows
Tumbling vs Sliding vs Session windows

Tumbling windows are fixed-size, non-overlapping — for example, every 5 minutes. Sliding windows are fixed-size but overlapping, controlled by a window size and a slide interval, useful for moving averages. Session windows group events by activity gaps — a new window starts after a period of inactivity, commonly used for user session analysis.

Lambda Architecture in practice

T-1 Offline + T Real-time — what goes wrong

Running batch (T-1) and streaming (T, today) pipelines side by side is the most common Flink production pattern, and the most common source of bugs.

Requires internet. Open in Chrome for audio.
Why this pattern exists
The setup

Most companies run two parallel pipelines for the same business metric. A T-1 batch job, usually Spark or Hive, runs overnight on yesterday's complete data and produces the authoritative, fully accurate numbers. A T real-time job, usually Flink, processes today's data as it streams in, giving dashboards a live, approximate view before the day closes. The next morning, the T-1 job recalculates and the dashboard switches to the accurate number.

Once T-1 completes for a date, serving switches exclusively to the batch table for that date
Typical dual pipeline
T-1Spark batch job runs at 1am, reads yesterday's complete data, writes authoritative tables
TFlink streaming job runs continuously, processes today's events as they arrive
MergeDashboard reads T-1 for past days, T real-time for today only, switches at midnight rollover
What goes wrong — the core problems
Problem 1 — Numbers don't match (caliber mismatch)

The single most common production incident. The real-time number for today and the batch number for the same day, computed the next morning, disagree, sometimes by a lot. This happens because the two jobs were written independently and subtly diverge in business logic — different filter conditions, different join keys, different handling of null values, or different definitions of what counts as a valid transaction.

Problem 2 — Late-arriving data

Real-time pipelines see events as they happen, but some events arrive late — a mobile app that was offline and syncs hours later, a retry from a failed upstream service, or a delayed message in a queue. The Flink job has already closed its time window and emitted a result by the time the late event shows up. The batch job the next day, reading the complete dataset, includes that late event and produces a different, usually higher, number. This is the single biggest source of T-1 versus T discrepancy.

Problem 3 — Duplicate counting across the boundary

At the moment T-1 data becomes available, typically just after midnight, both the leftover real-time stream for yesterday and the fresh batch job may briefly both be writing to overlapping time ranges. Without a clean cutover mechanism, a transaction near midnight can get counted once by the streaming job and once by the batch job, inflating the metric for a short window.

Problem 4 — Upstream data correction

Source systems sometimes correct data after the fact — a refunded transaction, a corrected currency conversion rate, a manually fixed data entry error. The batch job re-reads the corrected source data each night, so T-1 numbers can change retroactively. The real-time pipeline already processed and emitted the original, uncorrected value and has no mechanism to retroactively fix it unless explicitly designed to.

Problem 5 — Watermark too aggressive or too lenient

If the watermark delay is set too short, Flink closes windows before slow events arrive, causing systematic undercounting that the batch job will not reproduce. If the watermark delay is set too long, results lag further behind real time than the business requires, undermining the entire purpose of having a real-time pipeline. Tuning this delay requires understanding the actual distribution of event lateness in production, not guessing.

How to fix it
Fix 1 — Shared business logic

The most durable fix is to ensure the filter conditions, join logic, and aggregation rules are written once and shared between the batch and streaming jobs, not reimplemented twice. Some teams achieve this with a shared SQL definition executed through both Spark SQL and Flink SQL, since both support ANSI-like SQL syntax. This eliminates an entire category of caliber mismatch bugs at the source.

Fix 2 — Allowed lateness + side output

Flink lets you configure allowed lateness on a window — events arriving within this grace period still update the result and re-emit a corrected value. Events arriving after even the allowed lateness window can be routed to a side output stream instead of being silently dropped, so they can be inspected, logged, or merged back into the batch layer.

DataStream<Event> result = input .keyBy(Event::getUserId) .window(TumblingEventTimeWindows.of(Time.minutes(5))) .allowedLateness(Time.minutes(10)) .sideOutputLateData(lateOutputTag) .sum("amount");
Fix 3 — Clean cutover at the day boundary

Define an explicit cutover point — the real-time table only ever serves data for the current, incomplete day. The moment the T-1 batch job successfully completes for a given date, the dashboard layer switches that date's source from real-time to batch and never reads real-time data for that date again. This is typically implemented as a metadata flag or partition-ready marker that downstream consumers check before querying.

Fix 4 — Automated reconciliation job

Run a daily reconciliation check that compares the real-time pipeline's final number for yesterday against the batch pipeline's number for the same day, and alerts if the discrepancy exceeds a defined threshold, for example more than 0.5%. This does not prevent discrepancies but catches them immediately rather than letting them silently erode trust in the dashboards over weeks.

Production Issues

Common Flink pitfalls

Issues that show up once a job runs at real production scale, beyond the reconciliation problems above.

Requires internet. Open in Chrome for audio.
State State size grows unbounded over time ▼
Problem
Keyed state, such as a per-user counter, keeps growing forever if there is no expiration logic. Over months, RocksDB state can grow to hundreds of gigabytes, slowing checkpoints and eventually causing disk pressure on task managers.
Solution
Set a State TTL on the state descriptor so Flink automatically clears entries that haven't been updated within a configured window, for example 7 days for a user active today flag. Combine this with periodic checkpoint size monitoring to catch unexpected growth early.
StateTtlConfig ttlConfig = StateTtlConfig .newBuilder(Time.days(7)) .setUpdateType(StateTtlConfig.UpdateType.OnCreateAndWrite) .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired) .build();
Backpressure Job slows down, backpressure spreads upstream ▼
Problem
A slow downstream operator, often a sink writing to an external system like a database, cannot keep up with the input rate. Flink's flow control propagates this slowness backward through the entire pipeline, eventually causing the source to throttle reading from Kafka, increasing consumer lag.
Solution
Identify the bottleneck operator using Flink's web UI backpressure monitor. Common fixes include increasing parallelism on the slow operator, batching writes to the external sink instead of writing per-record, or switching to an async I/O operator so the sink doesn't block the main processing thread while waiting on network calls.
Data skew A few hot keys overload specific subtasks ▼
Problem
When keyBy is used for partitioning and one key, such as a particularly active merchant, accounts for a disproportionate share of traffic, that key's parallel subtask becomes a bottleneck while other subtasks sit idle. This is the streaming equivalent of the Spark data skew problem.
Solution
Apply the same salting technique used in Spark, append a random suffix to the hot key to split it across multiple subtasks for a local pre-aggregation stage, then merge the partial results in a second keyBy on the original key. Flink also supports rebalance and rescale operators to manually redistribute load.
Checkpoint Checkpoints keep timing out or failing ▼
Problem
As state grows large, the time required to snapshot it to durable storage increases. If a checkpoint takes longer than the configured checkpoint interval, checkpoints start queuing up and eventually fail, meaning the job loses its fault-tolerance guarantee.
Solution
Switch to incremental checkpoints with RocksDB, which only persist the delta since the last checkpoint instead of the full state, dramatically reducing checkpoint time for large state. Increase the checkpoint interval if sub-second recovery isn't required, and enable unaligned checkpoints so checkpoint barriers don't get stuck behind backpressure.
Interview Prep

Common Flink interview questions

Frequently asked in Data Engineer interviews, especially after mentioning T-1/T dual pipelines.

Requires internet. Open in Chrome for audio.
⚖️
Why do real-time and batch numbers for the same day disagree, and how do you fix it?
Reconciliation
▼

The most common cause is late-arriving data — the streaming job has already closed its window and emitted a result before some events arrive, while the batch job the next morning reads the complete dataset including those late events. The second most common cause is caliber mismatch — the filter conditions or join logic were implemented separately and subtly diverge between the two pipelines. The fix is twofold: use allowed lateness with a side output in Flink to capture late events instead of silently dropping them, and share the core business logic definition between both pipelines, ideally through a single SQL definition executed by both engines, rather than maintaining two independent implementations.

⏱️
How do you decide the watermark delay?
Watermark
▼

It should be based on measured data, not a guess. Look at the historical distribution of event lateness in production, for example what percentage of events arrive within 1 minute, 5 minutes, or 30 minutes of their event time. Choose a delay that covers a high percentile, such as the 99th percentile of lateness, balanced against how much result latency the business can tolerate. A financial dashboard that needs near-real-time numbers might accept a 1-minute delay and handle the remaining late events through allowed lateness and reconciliation; a less time-sensitive metric can use a longer, safer delay.

🔁
How does Flink guarantee exactly-once processing end to end?
Fault tolerance
▼

Internally, exactly-once state consistency is achieved through periodic checkpoints using the Chandy-Lamport snapshot algorithm. End-to-end exactly-once, meaning the output written to an external sink, additionally requires the sink to support transactions. Flink uses a two-phase commit protocol — the sink pre-commits data tied to a checkpoint, and only fully commits it once that checkpoint is confirmed complete. On failure and restart, any pre-committed but not fully committed transaction is rolled back, so no duplicate or partial writes reach the external system. Kafka, with its transactional producer API, is a common sink that supports this.

🆚
Flink vs Spark Structured Streaming — when would you choose Flink?
Comparison
▼

Spark Structured Streaming processes data in micro-batches — even at its smallest configurable interval, there is inherent latency because it waits to accumulate a batch before processing. Flink is a true event-at-a-time streaming engine, giving lower end-to-end latency, typically sub-second versus Spark's seconds-level latency. Flink also has a more mature and flexible state and windowing API, including session windows and custom triggers. I would choose Flink when latency requirements are strict, such as fraud detection or real-time alerting, and choose Spark Structured Streaming when the team already has deep Spark expertise and near-real-time, rather than true real-time, is acceptable.

🧮
How would you design a system that shows real-time numbers today but switches to accurate batch numbers tomorrow?
System design
▼

This is the classic Lambda architecture. A Flink job continuously processes today's stream and writes to a real-time serving table, partitioned by date. A nightly Spark batch job processes the complete previous day's data and writes to a separate, authoritative batch table for that same date. The serving layer checks a metadata flag indicating whether the batch job has completed for a given date. If not complete, it reads from the real-time table; once the batch job signals completion, all future reads for that date come exclusively from the batch table, and the real-time table's data for that date can be safely cleaned up. This guarantees every date eventually converges to one authoritative number while still providing live visibility during the day.

📈
How do you debug a Flink job that's falling behind (high consumer lag)?
Debugging
▼

Start with the Flink web UI's backpressure monitor to identify which operator is the bottleneck. Check whether that operator has sufficient parallelism relative to the source's parallelism. Look for data skew using subtask-level metrics — if one subtask is processing far more records than others, a hot key is likely the cause. If the bottleneck is a sink writing to an external system, check whether writes are happening one record at a time instead of batched, and whether the external system itself is the actual constraint rather than Flink.
